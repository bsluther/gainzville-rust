# UI Architecture Decisions

## Platform Targeting

### `#[cfg]` for input modality, CSS for layout

- **Input modality differences** (touch vs. pointer, one-handed reachability) → `#[cfg(feature = "mobile")]`
- **Spatial arrangement of the same content at different widths** → CSS
- **Different navigation containers** (tab bar vs. sidebar) → `#[cfg]`
- **Runtime viewport-width-dependent behavior** → `use_media_query` only if the above don't apply (no current cases)

```rust
#[cfg(feature = "mobile")]
fn EntryFieldEditor(/* ... */) -> Element {
    rsx! { BottomSheet { /* ... */ } }
}

#[cfg(not(feature = "mobile"))]
fn EntryFieldEditor(/* ... */) -> Element {
    rsx! { Popover { /* ... */ } }
}
```

### No runtime responsive rendering

No `use_media_query`, `use_breakpoint`, or runtime viewport-driven component swapping. Desktop components may be CSS-responsive (e.g., nav collapsing at narrow widths) but do not become mobile components. A `use_media_query` hook via `document::eval()` + `window.matchMedia()` is ~20 lines if a genuine runtime case surfaces later.

## Component Library

### dioxus-components (shadcn model)

The project uses [DioxusLabs/components](https://github.com/DioxusLabs/components), a shadcn-inspired UI library for Dioxus. It follows the same copy-paste model as shadcn: components are generated into the project and become project-owned code.

**Two-tier architecture:**

- **`dioxus-primitives`** (external crate, in `Cargo.toml`) — Unstyled, accessible interaction logic: keyboard navigation, ARIA, WAI-ARIA compliance. Provides the behavioral foundation for complex components (dialog, popover, calendar, dropdown menu, context menu). Do not modify.
- **Copied component files** (`src/components/<name>/component.rs` + `style.css` + `mod.rs`) — Styled wrappers around the primitives, generated by `dx components add <name>`. These files are project code, fully editable.

Simpler components (Button, Input) wrap native HTML elements directly and don't use `dioxus-primitives` at all. More complex ones (Sheet, Calendar, Popover, ContextMenu, DropdownMenu) import from it for interaction logic.

**Currently used components:** button, calendar, context_menu, dropdown_menu, input, popover, sheet.

**Adding a component:** `dx components add <name>` copies `component.rs`, `style.css`, and `mod.rs` into `src/components/<name>/`.

### Customization rule

The copied components in `src/components/<name>/` should remain generic — no GV-specific logic, no hardcoded assumptions about a particular view or use case. This keeps them reusable across the app, matching how shadcn components work.

For GV-specific needs:
1. **Pass props / children / CSS classes from the call site** — preferred for one-off or view-specific use.
2. **Create a named GV wrapper component** (e.g., `EntryFieldPopover`, `LibrarySheet`) — for patterns that repeat across the app and carry enough GV-specific behavior to warrant a name.

### Token alignment (pending)

The copied `style.css` files use `dx-components-theme.css` variables (`--primary-color`, `--secondary-color-*`) rather than GV tokens. These should be progressively replaced with `--gv-neutral-*` tokens for consistent theming. `assets/dx-components-theme.css` is the upstream theme — do not edit it directly; migrate by updating individual `style.css` files to reference `--gv-neutral-*` instead.

## Styling

### Concerns and approaches

| Concern | Approach |
|---|---|
| Design tokens (color, spacing scale) | CSS variables in `tokens.css` |
| Visual treatment (color, typography, borders referencing tokens) | CSS class |
| Stateful / contextual styles (hover, selected, variant) | CSS with data attributes + nesting |
| Structural layout one-offs (flex, gap, sizing that needs no name) | Tailwind utilities inline in RSX |
| Reusable patterns | **Component**, not CSS class — gives types, props, documentation |

**The key distinction for reuse:** prefer a Dioxus component over a CSS class whenever there is more than one sub-element or any logic. A class is appropriate for a simple atomic modifier with no children. When in doubt, use a component.

### Token system

CSS variables are loaded in three layers:

1. **`assets/dx-components-theme.css`** — upstream `dioxus-primitives` library theme (`--primary-color`, `--secondary-color-*`, `--focused-border-color`, etc.). Do not edit. GV tokens may override these.
2. **`assets/styling/tokens.css`** — GV design tokens (`--gv-neutral-*`, `--spacing-*`, `--gv-logged-blue`). This is the source of truth for app colors and spacing.
3. **Component-local tokens** — declared in `:root` at the top of a component's CSS file with a namespaced prefix (e.g., `--entry-text-color`, `--library-browser-width`). Aliases into layer 2 tokens.

All color and spacing references in CSS files must trace back to a token. No hardcoded hex values or bare pixel values for colors/spacing.

### File organization

- **`assets/styling/`** — global CSS only: `tokens.css`, `main.css`. Nothing component- or view-specific lives here.
- **`src/components/<name>.css`** — CSS for a single-file component, sibling to its `.rs` file (e.g., `entry_view.css` next to `entry_view.rs`). Loaded via `document::Link { href: asset!("./entry_view.css") }`.
- **`src/components/<name>/style.css`** — CSS for a multi-file component (folder-based, already used by `button/`, `input/`, `sheet/`). Loaded the same way.
- **`src/views/<name>.css`** — CSS for a view, sibling to its `.rs` file (e.g., `library.css` next to `library.rs`). Loaded in the view's top-level component.

### Tailwind v4 `@theme` as color source of truth

GV color values are defined in `dx-app/tailwind.css` inside an `@theme` block, not in `tokens.css`. This makes them first-class Tailwind utilities (`text-gv-neutral-400`, `bg-gv-neutral-900`, `border-gv-neutral-800/50`, etc.) usable inline in RSX alongside layout utilities.

`@theme` must hold concrete color values (not references to other CSS vars) because Tailwind resolves the values at build time to support opacity modifiers (`/50`, `/75`). Defining `--color-gv-neutral-400: var(--something)` in `@theme` would break opacity modifiers.

`tokens.css` then aliases back so CSS class files can continue using the `--gv-neutral-*` naming:

```css
/* tailwind.css — source of truth */
@theme {
    --color-gv-neutral-400: rgb(163, 163, 163);
}

/* tokens.css — alias for CSS file usage */
:root {
    --gv-neutral-400: var(--color-gv-neutral-400);
}
```

`assets/tailwind.css` is compiled output generated by `dx serve`/`dx build` — do not edit it directly.

### Spacing tokens

`--spacing-sm/md/lg/xl` in `tokens.css` are aliases to Tailwind's `--spacing` base unit (`0.25rem`), keeping CSS files automatically in sync with Tailwind utilities:

| `tokens.css` | Tailwind equivalent |
|---|---|
| `--spacing-sm` = `calc(var(--spacing) * 1)` | `p-1`, `gap-1` … |
| `--spacing-md` = `calc(var(--spacing) * 2)` | `p-2`, `gap-2` … |
| `--spacing-lg` = `calc(var(--spacing) * 3)` | `p-3`, `gap-3` … |
| `--spacing-xl` = `calc(var(--spacing) * 6)` | `p-6`, `gap-6` … |
