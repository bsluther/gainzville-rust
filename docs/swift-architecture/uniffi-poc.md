# UniFFI: Rust Core → Swift PoC Research

## Goal

Build a Swift iOS/macOS app that calls directly into `gv_core` + `gv_sqlite`, reusing 100% of the
domain/business logic while using native Swift for UI. UniFFI is the chosen bridge.

---

## Existing Rust API Surface

**`gv_sqlite::SqliteClient`** — the main entry point:
- `SqliteClient::init(db_path: &str) -> Result<Self>` (async)
- `run_action(&self, action: Action) -> Result<()>` (async)
- `stream_activities()`, `stream_entries()`, etc. (async streams via tokio broadcast)

All operations are `async`/`.await`. sqlx is configured with `runtime-tokio`. tokio `full` is a
workspace-level dependency.

---

## Root Cause: Previous Threading Problems

Two likely causes, either or both:

### A. No active tokio runtime on the Swift-called thread

UniFFI's `async_runtime="tokio"` uses `tokio::runtime::Handle::current()` internally, which
**panics** if no runtime is running on the current thread. Swift calls into Rust from its own thread
pool — no tokio context is present.

### B. UniFFI Issue #2576 (open as of June 2025)

The `async_runtime="tokio"` attribute is **silently ignored** when the exported function comes from
an async *trait method* (i.e., exposed via `Arc<dyn Trait>`). The generated bridging code lacks the
`async-compat` wrapper needed for tokio. This doesn't apply when exposing concrete structs directly.

## Recommended Architecture

### New crate: `gv-ffi`

A thin FFI shim added to the workspace. **Zero changes to `gv_core` or `gv_sqlite`.**

```
gv-ffi/
  Cargo.toml   # crate-type = ["cdylib", "staticlib"]
  src/
    lib.rs     # uniffi::setup_scaffolding!(); mod client; mod types;
    client.rs  # GvClient with #[derive(uniffi::Object)]
    types.rs   # FFI-safe mirror types (FfiActivity, FfiAction, FfiError)
```

```toml
# gv-ffi/Cargo.toml
[dependencies]
gv_sqlite = { path = "../sqlite" }
uniffi = { version = "0.28", features = ["proc-macro"] }
tokio = { workspace = true }
once_cell = "1"

[build-dependencies]
uniffi = { version = "0.28", features = ["build"] }

[lib]
crate-type = ["cdylib", "staticlib"]
```

### Threading Solution: Static Tokio Runtime + Sync Wrappers

Avoid async UniFFI plumbing entirely. Expose **synchronous** functions backed by a static runtime:

```rust
static RUNTIME: Lazy<Runtime> = Lazy::new(|| {
    tokio::runtime::Builder::new_multi_thread()
        .enable_all()
        .build()
        .unwrap()
});

#[derive(uniffi::Object)]
pub struct GvClient { inner: Arc<SqliteClient> }

#[uniffi::export]
impl GvClient {
    #[uniffi::constructor]
    pub fn new(db_path: String) -> Arc<Self> {
        let client = RUNTIME.block_on(SqliteClient::init(&db_path)).unwrap();
        Arc::new(Self { inner: Arc::new(client) })
    }

    pub fn get_activities(&self) -> Vec<FfiActivity> {
        RUNTIME.block_on(async { /* Reader queries via pool */ })
    }

    pub fn run_action(&self, action: FfiAction) -> Result<(), FfiError> {
        RUNTIME.block_on(self.inner.run_action(action.into())).map_err(Into::into)
    }
}
```

**Why sync instead of async UniFFI:**
- Eliminates Issue #2576 entirely
- No risk of panics from missing runtime context
- Swift can dispatch to a background thread trivially (`Task.detached`, `DispatchQueue.global`)
- SQLite operations are fast; sync wrappers are fine for a PoC

**Streams → snapshots:** UniFFI doesn't support Rust streams. Replace `stream_activities()` etc.
with one-shot snapshot reads. Change notification can be a follow-on (callback interface or polling).

### Minimal PoC FFI Surface

1. `GvClient::new(db_path: String) -> GvClient`
2. `GvClient::get_activities() -> Vec<FfiActivity>`
3. `GvClient::run_action(action: FfiAction) -> Result<(), FfiError>`

`FfiAction` starts with just `CreateActivity { name: String }` to validate the round-trip.

### Swift Side

```swift
import GvFfi  // auto-generated by uniffi-bindgen

let dbPath = FileManager.default
    .urls(for: .documentDirectory, in: .userDomainMask).first!
    .appendingPathComponent("gv.db").path

let client = GvClient(dbPath: dbPath)
try client.runAction(action: .createActivity(name: "Back Squat"))
let activities = client.getActivities()  // → [FfiActivity]
```

## Implementation Steps

1. **Add `gv-ffi` crate** to workspace (`Cargo.toml` members)
2. **Define FFI types** — `FfiActivity`, `FfiAction`, `FfiError` as UniFFI records/enums (mirror,
   don't re-export `gv_core` types directly)
3. **Implement `GvClient`** with static runtime and sync wrappers
4. **Write `build.rs`** for uniffi scaffolding generation
5. **Generate Swift bindings** via `uniffi-bindgen generate --language swift`
6. **Build xcframework:**
   ```sh
   cargo build --release --target aarch64-apple-ios
   cargo build --release --target aarch64-apple-ios-sim
   xcodebuild -create-xcframework ...
   ```
7. **Create minimal Swift app** — import the xcframework, run the round-trip test

### Tooling Options

- **`cargo-swift`** — automates steps 5–6, produces a ready-to-import Swift package.
  Try this first: `cargo install cargo-swift`
- **Manual** — `uniffi-bindgen` + `xcodebuild -create-xcframework`

## Risks & Open Questions

| Risk | Mitigation |
|------|-----------|
| sqlx compile-time verification requires live Postgres | Build `gv-ffi` with `SQLX_OFFLINE=true` + offline query cache |
| Static runtime init failure | Use `GvClient::try_new()` returning `Result`; fail fast with a clear error |
| xcframework toolchain complexity | `cargo-swift` abstracts most of this |
| Rust edition 2024 + UniFFI 0.28 compatibility | Verify before starting; may need a newer UniFFI release |
| iOS SQLite path | Use app sandbox Documents directory, not hardcoded paths |

## Verification

End-to-end PoC validation:
1. Swift creates `GvClient` with a temp DB path
2. Calls `run_action(.createActivity(name: "Back Squat"))`
3. Calls `get_activities()` → receives `[FfiActivity]` containing the new activity
4. Runs on both macOS and iOS Simulator targets

Full stack: Swift → UniFFI → static tokio runtime → `gv_sqlite` → SQLite → `gv_core` domain logic.
